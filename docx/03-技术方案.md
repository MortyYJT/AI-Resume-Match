# 技术方案

这份文档说明当前技术方向。第一期只用最少技术跑通 Demo。

## 当前技术选择

| 模块 | 选择 | 第一期开不开 |
| --- | --- | --- |
| 后端 | Java + Spring Boot | 开 |
| 前端 | React 或 Vue，前端自选 | 开 |
| 数据库 | MySQL | 暂不开 |
| AI | 大模型 API | 暂不开 |
| PDF 解析 | PDFBox | 暂不开 |
| 部署 | 本地先跑 | 开 |

## 第一期后端只做什么

后端只需要提供两个接口：

```text
GET /api/health
POST /api/matches/mock
```

`/api/health` 用来确认后端是否启动成功。

`/api/matches/mock` 用来返回一份固定的匹配报告，让前端先把页面和联调跑通。

## 推荐后端结构

第一期可以先保持简单：

```text
com.airesumematch
  common
    ApiResponse
  health
    HealthController
  match
    MatchController
    MockMatchService
```

后续做大了再拆更多模块。

## 第一阶段不接 MySQL 的原因

第一期目标是先验证前后端协作，不是验证数据库。

先不接 MySQL 可以减少这些问题：

- 本地数据库安装和连接问题
- 表结构争论
- 数据迁移问题
- 后端还没跑通就被配置卡住

等 mock Demo 跑通后，再接 MySQL 更稳。

## 第一阶段不接真实 AI 的原因

真实 AI 会带来这些变量：

- API key
- 费用
- 网络问题
- 返回格式不稳定
- 调用速度慢

第一期先用 mock 数据，等页面和接口通了，再把 mock 替换成真实 AI。

## 后续技术引入顺序

建议顺序：

1. Spring Boot 基础接口
2. 前后端联调
3. PDF 上传
4. PDF 文本解析
5. 大模型 API
6. MySQL
7. 登录注册
8. Docker 和云端部署

## 暂时不要碰

- Redis
- RabbitMQ
- Elasticsearch
- 对象存储
- 微服务
- Kubernetes

这些等真的遇到问题再学。
