# 第一期计划

第一期只做一个能演示的 Demo 骨架。

## 一句话目标

```text
前端页面 -> 调后端 mock 接口 -> 后端返回固定匹配报告 -> 前端展示报告
```

## 为什么第一期这么做

我们现在是 3 人小团队，而且主要用 vibe coding。

第一期先不碰数据库、PDF、真实 AI，是为了先确认：

- 后端能跑起来
- 前端能跑起来
- 前后端能联调
- 页面能展示一份完整报告

这些跑通之后，再接真实能力会更稳。

## 第一期要做

- Spring Boot 后端能本地启动
- 前端项目能本地启动
- 后端提供 `/api/health`
- 后端提供 `/api/matches/mock`
- 前端做一个 Demo 页面
- 前端能输入简历内容和 JD
- 前端能展示匹配报告

## 第一期不做

- 登录注册
- MySQL
- PDF 上传
- PDF 解析
- 真实大模型 API
- 历史记录
- Redis
- Docker
- 云端部署

## 分工

| 成员 | 第一期任务 |
| --- | --- |
| 后端 A | 初始化后端、写 health 接口、写 mock 分析接口、推进联调 |
| 后端 B | 本地跑通后端、熟悉代码结构、后续准备接 PDF 或数据库 |
| 前端 A | 初始化前端、做 Demo 页面、调用 mock 接口、展示报告 |
| 产品顾问 | 看 Demo 流程是否合理，帮忙砍需求 |

## 接口

第一期只实现两个接口。

### 健康检查

```text
GET /api/health
```

### Mock 匹配分析

```text
POST /api/matches/mock
```

详细字段见 [接口设计](04-接口设计.md)。

## 页面

第一期前端只需要一个页面。

页面包含：

- 简历内容输入框
- 岗位 JD 输入框
- 开始分析按钮
- 匹配分数
- 优势
- 短板
- 简历优化建议
- 面试准备建议

## 任务清单

| 优先级 | 任务 | 负责人 | 完成标准 |
| --- | --- | --- | --- |
| P0 | 初始化 Spring Boot 后端 | 后端 A | 后端本地能启动 |
| P0 | 写 `/api/health` | 后端 A | 能访问并返回 success |
| P0 | 写 `/api/matches/mock` | 后端 A | 能返回固定匹配报告 |
| P0 | 初始化前端项目 | 前端 A | 前端本地能打开 |
| P0 | 做 Demo 页面 | 前端 A | 能输入简历内容和 JD |
| P0 | 前端调用 mock 接口 | 前端 A | 点击分析后展示报告 |
| P1 | 本地跑通后端 | 后端 B | 能启动后端并访问 health |
| P1 | 熟悉后端结构 | 后端 B | 能说明接口代码在哪里 |

## 建议节奏

```text
Day 1：后端和前端项目都跑起来
Day 2：后端 mock 接口 + 前端 Demo 页面
Day 3：前后端联调 + 修问题
```

## 完成标准

本地打开前端页面，输入简历内容和岗位 JD，点击分析，页面展示匹配分数、优势、短板和建议。

能做到这个，第一期完成。

## 内部同步摘要

```text
第一期我们先做 Demo 骨架，不做复杂功能。

目标：
前端页面 -> 调后端 mock 接口 -> 后端返回固定匹配报告 -> 前端展示报告。

第一期不做：
登录、MySQL、PDF 解析、真实 AI、历史记录、Redis、Docker、部署。

分工：
- 后端 A：初始化 Spring Boot，写 health 接口和 mock 分析接口
- 后端 B：本地跑通后端，熟悉项目结构
- 前端 A：初始化前端，做 Demo 页面，调用 mock 接口展示报告
- 产品顾问：看 Demo 流程是否合理，帮忙砍需求

验收标准：
本地打开前端页面，输入简历内容和 JD，点击分析，页面展示匹配分数、优势、短板和建议。
```
