# 开发规范

这份文档不是为了限制大家，而是为了避免 vibe coding 把项目写乱。

## 最重要的规则

每次只做一个小功能。

例如：

- 这次只写 health 接口。
- 这次只写 mock 分析接口。
- 这次只做结果展示页面。

不要一次让 AI 生成整个系统。

## Git 使用

第一期可以简单一点：

```text
main：稳定代码
feature/功能名：开发功能
```

分支示例：

```text
feature/backend-health
feature/mock-match-api
feature/demo-page
```

提交信息尽量写清楚：

```text
feat: add health api
feat: add mock match api
feat: add demo page
fix: fix api request error
docs: update phase 1 plan
```

## 后端规则

- 接口返回统一格式：`code`、`message`、`data`。
- Controller 只负责接请求和返回结果。
- 复杂逻辑放到 Service。
- 接口改了，要同步改 `04-接口设计.md`。
- 第一期开 mock 数据，不要提前接数据库。

## 前端规则

- 第一页先做完整流程，不追求页面很复杂。
- 先能调用接口，再优化样式。
- 接口地址、字段名要和 `04-接口设计.md` 一致。
- 报错时页面要能看出来，不要静默失败。

## Vibe Coding 规则

让 AI 写代码前，先说清楚 4 件事：

```text
我要做什么功能？
输入是什么？
输出是什么？
不要做什么？
```

示例：

```text
请帮我写 Spring Boot 的 /api/matches/mock 接口。
输入是 resumeText 和 jdText。
输出是固定的匹配报告 JSON。
不要接数据库，不要调用真实 AI。
```

AI 写完后必须做两件事：

1. 本地运行。
2. 用浏览器或接口工具验证。

## 暂时不要引入

- Redis
- RabbitMQ
- Docker
- 登录鉴权
- 对象存储
- 复杂权限

引入新技术前，先在群里说明为什么现在需要。
